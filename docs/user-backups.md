# Пользовательские бэкапы и восстановление

## Цель

Дать авторизованному пользователю возможность самостоятельно скачать и
проверить переносимую резервную копию своих данных Planner.

Первая версия сознательно ограничена личным пространством пользователя
(`personal workspace`). Общие пространства требуют отдельных правил по правам
владельца, приватности участников, конфликтам восстановления и строкам других
пользователей.

## Текущее состояние проекта

- Инфраструктурный бэкап уже есть: `npm run db:backup` запускает
  `pg_dump --format=custom` для всей Postgres-базы. Он используется перед
  production-миграциями.
- Инфраструктурный бэкап не является пользовательской функцией: для него нужны
  админские доступы к базе, он включает все tenant-данные и сам по себе не
  покрывает локальные файлы ассетов.
- Web-клиент считает backend/Postgres источником истины. IndexedDB используется
  для server-state cache, offline snapshots и очереди offline-мутаций.
- Загруженные иконки и аватары живут не только в Postgres: файловые данные
  находятся в `API_ICON_ASSET_DIR`.

## Уже реализовано

Статус на 2026-07-09: реализована первая безопасная часть пользовательских
бэкапов.

- Добавлен общий контракт формата архива в `packages/contracts/src/backup.ts`.
- Добавлен backend-модуль `apps/api/src/modules/backups`.
- Реализован экспорт личного пространства:
  `GET /api/v1/backups/export`.
- Реализована проверка импортируемого файла без применения изменений:
  `POST /api/v1/backups/import/preview`.
- API отдаёт архив как JSON attachment с версионированным форматом
  `planner.user-backup`.
- Экспорт работает только для авторизованного пользователя и только для
  `personal workspace`.
- В `/more` добавлен UI-блок “Резервные копии”: скачать копию и проверить файл.
- В Android-приложении экспорт сохраняется нативно в `Загрузки/Chaotika`,
  чтобы файл был виден через системный файловый менеджер. В мобильном браузере
  используется системное меню сохранения/поделиться, если оно доступно. В
  desktop-браузере остаётся штатный сценарий download/save dialog.
- UI показывает прогресс подготовки, скачивания, сохранения, чтения файла и
  preview-проверки архива.
- Выбор файла для проверки на мобильных устройствах использует расширенный
  фильтр типов, чтобы Android file picker не скрывал JSON-файлы бэкапа.
- Добавлены unit-тесты сервиса бэкапов и smoke-проверка экспорта на реальной
  Postgres-базе.
- Экспорт задач ограничен пользовательски полезным объёмом: удалённые задачи не
  включаются, выполненные задачи попадают только за последние 14 дней, а
  связанные time blocks, occurrences, attachments и chains берутся только для
  экспортируемых задач.
- Экспорт выполняется в `REPEATABLE READ READ ONLY` snapshot; ссылки daily plan
  нормализуются по набору фактически экспортированных задач.
- Preview импорта использует строгую схему формата v1, запрещает неизвестные
  таблицы и колонки, проверяет UUID, scope, дубликаты, незамкнутые ссылки и
  отсутствующие payload локальных ассетов.
- Глобальная emoji-библиотека, загруженная администратором приложения, исключена
  из пользовательского self-service backup. Она доступна всем пользователям и
  должна бэкапиться отдельно как часть системного/admin backup.

Сознательно не реализовано в первой версии:

- применение восстановления в базу;
- destructive replace-restore;
- восстановление shared workspace;
- импорт чужого workspace без правил remap/conflict handling;
- автоматическая запись файлов ассетов из архива обратно в `API_ICON_ASSET_DIR`.

## Объём MVP

MVP состоит из двух безопасных операций:

- скачать бэкап текущего личного пространства;
- проверить загруженный архив и показать пользователю, что в нём находится.

Применение восстановления откладывается до отдельного этапа, потому что оно
может менять пользовательские данные и требует транзакций, remap ID, правил
конфликтов и отдельного покрытия тестами.

## Формат архива

Формат архива: `planner.user-backup`, версия `1`.

Архив содержит:

- manifest: время экспорта, версия формата, версия приложения-источника;
- scope: `userId`, `workspaceId`, `workspaceKind`;
- выбранные domain-строки текущего пользователя и workspace;
- payload локальных profile/icon ассетов, на которые ссылаются экспортируемые
  строки.

## Данные в экспорте

В экспорт первой версии входят пользовательские и workspace-данные:

- профиль и настройки пользователя без auth-секретов и `app_role`;
- текущее личное пространство и membership-строка пользователя;
- задачи, цепочки задач, time blocks, occurrences, attachments;
- шаблоны задач и жизненные сферы;
- daily plans, chaos inbox и shopping items;
- зоны уборки, задачи уборки, состояния и история;
- привычки и записи привычек;
- self-care items, schedules, occurrences, completions, settings и detail
  tables;
- связанный файл аватара профиля.

Для задач действует отдельная политика отсечения истории:

- soft-deleted задачи исключаются из архива;
- задачи со статусом `done` включаются только если `completed_at` попадает в
  последние 14 дней на момент экспорта;
- открытые, активные, архивные и запланированные задачи без `deleted_at`
  включаются;
- task-domain дочерние строки экспортируются только для задач, прошедших этот
  фильтр.

## Данные вне экспорта

Пользовательский архив не должен включать секреты и runtime-only данные:

- password hashes, refresh tokens, reset tokens, OAuth authorization codes;
- MCP OAuth tokens и audit logs;
- push tokens, device sessions, sync cursors;
- outbox rows и обработанные integration jobs;
- сгенерированные notification delivery rows, которые можно пересоздать из
  task/self-care schedule state;
- системные self-care templates;
- глобальные `emoji_sets`, `emoji_assets` и связанные icon asset файлы
  администраторской emoji-библиотеки.

Preview импорта также предупреждает, если:

- строка архива ссылается на parent-строку, которой нет в архиве;
- строка архива ссылается на локальный файл, но payload этого файла не включён
  в `assets`;
- asset payload содержит дублирующийся путь или не совпадает с заявленным
  `byteLength`;
- base64 неканонический или magic bytes не соответствуют заявленному MIME;
- архив содержит дубликаты ID или строки другого user/workspace scope;
- daily plan содержит ссылки на задачи, отсутствующие в архиве.

## Следующий шаг

Следующий инженерный шаг: реализовать безопасное восстановление личного
пространства в режиме merge-restore.

Минимальный объём следующего этапа:

1. Добавить backend-операцию применения архива после успешного preview:
   `POST /api/v1/backups/import/restore`.
2. Выполнять restore внутри одной транзакции.
3. Поддержать только `personal workspace`.
4. Сначала вставлять отсутствующие строки, затем обновлять существующие только
   по явному правилу конфликта.
5. Не удалять локальные строки в merge-режиме.
6. Добавить remap небезопасных identity-полей при импорте в другого
   пользователя или workspace.
7. Записывать asset-файлы из архива обратно в `API_ICON_ASSET_DIR` только после
   успешной валидации.
8. После успешного restore инвалидировать локальные IndexedDB-кэши на клиенте.
9. Добавить Postgres integration tests на порядок вставки, foreign keys,
   конфликтные строки и rollback при ошибке.

Destructive replace-restore должен идти отдельным этапом после merge-restore.
Перед replace нужно создавать свежий server-side restore point и требовать
явное подтверждение в UI.

## Production и disaster recovery

Пользовательский self-service backup не заменяет production disaster recovery.
Для production всё равно нужны:

- регулярный `pg_dump` или provider snapshots;
- отдельный бэкап `API_ICON_ASSET_DIR`;
- хранение копий вне основного сервера;
- шифрование и retention policy;
- периодическая проверка `pg_restore` на одноразовой базе.
